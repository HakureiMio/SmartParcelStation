你现在修改的是 GitHub 仓库 `HakureiMio/SmartParcelStation`。请在当前项目结构基础上实现第一阶段真实硬件闭环：

员工微信小程序 → 局域网 HTTP 调用 smartparcel-gateway → gateway 通过 BLE 连接 nRF52810 智能寻物标签 → 控制标签 RGB LED 和蜂鸣器。

本次只实现“员工小程序控制网关连接和控制标签”的最小闭环，不要实现完整取件、门禁、server 云同步、生产 HTTPS、小程序正式发布域名配置等功能。

## 一、总体架构要求

目标链路：

```text
smartparcel-miniprogram 员工端
  -> wx.request 调用局域网 gateway local API
  -> smartparcel-gateway FastAPI local_api
  -> gateway BLE service
  -> nRF52810 标签 GATT Service
  -> RGB LED / buzzer 执行
```

保留现有 mock 能力，但新增真实 BLE 能力。通过环境变量选择：

```env
BLE_BACKEND=mock
BLE_BACKEND=real
```

默认仍然使用 `mock`，避免没有硬件时项目无法运行。真实硬件测试时切换为 `real`。

## 二、标签命名与编号规则

实体标签使用固定规范蓝牙名称，并可通过丝印/贴纸记录在标签外壳上：

```text
SPS-TAG-0001
SPS-TAG-0002
SPS-TAG-0003
```

规则：

```text
SPS-TAG-XXXX
```

其中 `XXXX` 是唯一序列号。第一阶段中，`tag_uid` 和 `tag_id` 都可以使用这个蓝牙名称，例如 `SPS-TAG-0001`。

网关连接/注册标签后，由 gateway 分配本地显示编号：

```text
001
002
003
```

员工小程序普通列表只显示：

```text
标签 001
标签 002
```

点开标签属性时显示：

```text
网关编号：标签 001
唯一序列号：SPS-TAG-0001
BLE 地址：xx:xx:xx:xx:xx:xx
状态：ONLINE / OFFLINE / CONNECTED / IDLE / ERROR
电池电压：xxxx mV
最后发现时间
最后连接时间
```

请在 gateway 的 `LocalTag` 模型中新增或复用字段来支持这个逻辑。现有字段包括 `tag_id`、`tag_uid`、`ble_address`、`status`、`battery_level`、`battery_mv`、`registered_at`、`last_seen_at`。请新增至少以下字段：

```python
local_no: int | None          # 网关本地编号，例如 1、2、3
display_name: str | None      # 例如 标签 001
ble_name: str | None          # 例如 SPS-TAG-0001
last_connected_at: datetime | None
```

同时更新 `gateway/db/init_db.py`，为已有 SQLite 数据库追加这些列。当前项目没有完整 Alembic，gateway 使用 `Base.metadata.create_all()` + `ALTER TABLE` 兼容旧库，请沿用这个风格。

## 标签命名与编号规则修正

请将“标签蓝牙名称”和“网关本地显示编号”分离。

### 1. 标签出厂蓝牙名称

标签固件广播的 BLE 设备名称不使用 `SPS-TAG-0001` 这种本地编号，而使用出厂信息格式：

```text
SPS-F01-20260610-0001
```

格式定义：

```text
SPS-{factory_code}-{production_date}-{serial_no}
```

示例：

```text
SPS-F01-20260610-0001
SPS-F01-20260610-0002
SPS-F02-20260611-0001
```

字段含义：

```text
SPS：SmartParcelStation 设备前缀
F01：工厂编号或生产批次来源
20260610：生产日期，YYYYMMDD
0001：当日/该批次生产序号
```

这个名称应作为标签的出厂唯一识别信息，可以通过丝印、贴纸或二维码记录在实体标签外壳上。第一阶段不要求网关或小程序修改 BLE 设备名称。

### 2. 网关本地编号

标签被某个 gateway 扫描并注册后，由 gateway 分配本地可读编号。

本地系统编码：

```text
SPS-TAG-0001
SPS-TAG-0002
SPS-TAG-0003
```

员工小程序普通列表显示：

```text
标签 001
标签 002
标签 003
```

点开标签属性页时显示：

```text
网关编号：标签 001
本地标签编码：SPS-TAG-0001
出厂唯一名称：SPS-F01-20260610-0001
BLE 地址：AA:BB:CC:DD:EE:FF
状态：ONLINE / OFFLINE / CONNECTED / RUNNING / ERROR
电池电压：xxxx mV
最后发现时间
最后连接时间
```

### 3. 数据字段建议

请在 gateway 的 `LocalTag` 中支持以下含义：

```python
tag_id: str                  # 网关本地标签编码，例如 SPS-TAG-0001
tag_uid: str                 # 出厂唯一名称，例如 SPS-F01-20260610-0001
ble_name: str | None         # 实际扫描到的 BLE 名称，例如 SPS-F01-20260610-0001
ble_address: str | None      # BLE MAC / address
local_no: int | None         # 网关本地序号，例如 1
display_name: str | None     # 小程序显示名，例如 标签 001
factory_code: str | None     # 可选，从 ble_name 解析，例如 F01
production_date: str | None  # 可选，从 ble_name 解析，例如 20260610
factory_serial: str | None   # 可选，从 ble_name 解析，例如 0001
last_connected_at: datetime | None
```

如果为了减少数据库改动，也可以先不拆 `factory_code / production_date / factory_serial`，第一阶段只保存完整 `tag_uid / ble_name` 即可。

### 4. 注册逻辑

`POST /local/tags/register-from-ble` 的请求改为：

```json
{
  "ble_name": "SPS-F01-20260610-0001",
  "ble_address": "AA:BB:CC:DD:EE:FF"
}
```

注册行为：

1. 校验 `ble_name` 是否符合：

```regex
^SPS-[A-Z0-9]{2,8}-[0-9]{8}-[0-9]{4,8}$
```

2. 如果 `tag_uid` 或 `ble_name` 已存在，则更新 `ble_address / last_seen_at`，不要重复创建标签。
3. 如果不存在，创建新标签。
4. 为新标签分配当前 gateway 内的下一个 `local_no`。
5. 生成：

```text
tag_id = SPS-TAG-{local_no:04d}
display_name = 标签 {local_no:03d}
tag_uid = ble_name
ble_name = ble_name
```

示例：

```text
扫描到 BLE 名称：SPS-F01-20260610-0001
注册后网关本地编码：SPS-TAG-0001
员工端显示：标签 001
详情页显示出厂唯一名称：SPS-F01-20260610-0001
```

### 5. BLE 扫描逻辑

gateway 扫描 BLE 时，不再筛选 `SPS-TAG-` 开头，而筛选出厂 BLE 名称：

```regex
^SPS-[A-Z0-9]{2,8}-[0-9]{8}-[0-9]{4,8}$
```

mock 扫描数据也改为：

```json
[
  {
    "ble_name": "SPS-F01-20260610-0001",
    "ble_address": "MOCK:TAG:0001",
    "rssi": -45
  }
]
```

### 6. 标签固件默认 BLE 名称

nRF52810 标签固件第一阶段默认 BLE 名称改为：

```conf
CONFIG_BT_DEVICE_NAME="SPS-F01-20260610-0001"
```

启动日志打印：

```text
BLE name: SPS-F01-20260610-0001
```

后续真实批量生产时，每个标签应烧录不同的出厂 BLE 名称，或通过 NVS/产测工具写入唯一出厂信息。本阶段先写死一个名称用于测试。

### 2. BLE 名称

第一阶段先把默认 BLE 名称写死为：

```text
SPS-TAG-0001
```

后续再做 NVS 动态配置。本次不要求实现手机/网关修改蓝牙名称。

启动日志中打印：

```text
BLE name: SPS-TAG-0001
```

### 3. 自定义 GATT Service

在 `src/ble_clip_service.c/.h` 中把 mock service 改成真实 GATT Service。

请定义固定 UUID，可以使用 128-bit UUID：

```text
SPS_TAG_SERVICE_UUID
SPS_TAG_CMD_WRITE_UUID
SPS_TAG_EVENT_NOTIFY_UUID
SPS_TAG_STATUS_READ_UUID
```

特征值：

1. `CMD_WRITE`

   * Write Without Response 或 Write
   * gateway 写入当前 `clip_protocol` 帧
   * 收到数据后调用现有 `g_cmd_handler(data, len)`

2. `EVENT_NOTIFY`

   * Notify
   * `ble_clip_service_send_event(data, len)` 通过 notify 发给 gateway
   * 如果没有订阅 notify，则至少写日志，不要崩溃

3. `STATUS_READ`

   * Read，可选
   * 返回一个简短状态结构，或返回最近一次状态帧
   * 如果实现困难，可以第一阶段返回固定文本或最近状态缓存

### 4. 保留现有命令协议

不要推翻 `clip_protocol.h/c`。继续使用：

```c
CLIP_CMD_PING
CLIP_CMD_WAKE_TAG
CLIP_CMD_STOP_ALERT
CLIP_CMD_SET_BINDING
CLIP_CMD_CLEAR_BINDING
CLIP_CMD_READ_STATUS
```

以及事件：

```c
CLIP_EVT_BOOT
CLIP_EVT_COMMAND_ACK
CLIP_EVT_STATUS_REPORT
CLIP_EVT_PONG
```

现有 `main.c` 已经能处理 `WAKE_TAG / STOP_ALERT / READ_STATUS`，请尽量复用，不要大改业务状态机。

### 5. RGB 和蜂鸣器测试要求

`WAKE_TAG` 后：

* RGB LED 开始寻物闪烁
* 蜂鸣器开始间歇鸣叫
* 30 秒后自动停止
* 收到 `STOP_ALERT` 立即关闭 RGB 和蜂鸣器

这个行为已存在，请保持兼容。

## 四、gateway 真实 BLE 服务


### 1. requirements

### 2. 新增 BLE 服务结构

新增目录：

```text
smartparcel-gateway/gateway/services/ble/
```

建议文件：

```text
__init__.py
protocol.py
base.py
mock.py
real.py
manager.py
```

要求：

### `protocol.py`

用 Python 实现与 `clip_protocol.c` 一致的帧构造和解析：

帧格式：

```text
[0] header = 0xA5
[1] cmd_or_evt
[2] payload_len
[3..] payload
[last] xor checksum
```

实现函数：

```python
build_command(cmd: int, payload: bytes = b"") -> bytes
parse_event(frame: bytes) -> dict
```

命令常量：

```python
CLIP_CMD_PING = 0x01
CLIP_CMD_WAKE_TAG = 0x02
CLIP_CMD_STOP_ALERT = 0x03
CLIP_CMD_SET_BINDING = 0x04
CLIP_CMD_CLEAR_BINDING = 0x05
CLIP_CMD_READ_STATUS = 0x06
```

事件常量也同步定义。

### `base.py`

定义抽象接口：

```python
class BleTagServiceBase:
    async def scan_tags(self, timeout_sec: float = 5.0) -> list[dict]: ...
    async def connect_tag(self, ble_address: str) -> dict: ...
    async def disconnect_tag(self, ble_address: str) -> dict: ...
    async def wake_tag(self, ble_address: str, color: str = "BLUE", duration_sec: int = 30) -> dict: ...
    async def stop_alert(self, ble_address: str) -> dict: ...
    async def read_status(self, ble_address: str) -> dict: ...
```

### `mock.py`

实现 mock 版本，用于无硬件开发。mock 扫描至少返回：

```json
[
  {
    "ble_name": "SPS-TAG-0001",
    "ble_address": "MOCK:TAG:0001",
    "rssi": -45
  }
]
```

### `real.py`

使用 `bleak` 实现真实 BLE：

* `scan_tags()`：扫描附近 BLE 设备，只返回名称以 `SPS-TAG-` 开头的设备。
* `connect_tag()`：连接指定 `ble_address`。
* `wake_tag()`：连接后向 `CMD_WRITE` characteristic 写入 `WAKE_TAG` 协议帧。
* `stop_alert()`：写入 `STOP_ALERT` 协议帧。
* `read_status()`：写入 `READ_STATUS`，尽量通过 notify 等待状态事件；如果第一阶段通知等待复杂，可以先返回 write 成功，并在后续补充通知解析。
* 保持连接可以第一阶段简化：每次命令短连接，执行后断开。后续再做连接池。
* 出错时返回结构化错误，不要直接让 API 崩溃。

返回示例：

```json
{
  "ok": true,
  "backend": "real",
  "ble_name": "SPS-TAG-0001",
  "ble_address": "AA:BB:CC:DD:EE:FF",
  "action": "wake",
  "message": "WAKE_TAG sent"
}
```

失败示例：

```json
{
  "ok": false,
  "backend": "real",
  "error": "device_not_found"
}
```

### `manager.py`

提供同步封装，供 FastAPI local_api 调用。因为 FastAPI 端目前是普通 def，也可以把 local_api 改成 async def。请选择代码最清晰的方式。
